## Findings

### F-R9-1 — Backup tarball is a header-only file: closing the gz writer never finalizes a valid archive
- **Category**: correctness / data-loss
- **Confidence**: high
- **Location**: port.rs:62-91 (`write_backup`)
- **Today**: `tar::Builder::new(gz)` consumes `GzEncoder<File>`. After `append_*`, code does `tar.into_inner().and_then(|gz| gz.finish())`. `Builder::into_inner()` returns inner writer **without** writing two 512-byte zero blocks tar mandates as end-of-archive. Resulting `port-<ts>.tar.gz` is valid gzip but truncated tar — `tar -tzf` prints "Unexpected EOF in archive". `bsdtar` on macOS and Windows' Tar.exe both reject. No fsync issued before tarball treated as durable.
- **Proposed**: Call `tar.finish()` (writes EOF marker) before `into_inner()`. Then `gz.finish()`, then `file.sync_all()`. Add post-write integrity probe via `flate2::read::GzDecoder + tar::Archive`. Fsync parent directory on Unix.
- **Side effects**: Verification re-reads the file (~5-50ms). Acceptable.

### F-R9-2 — Port writes are not atomic: a crash between backup and rewrite leaves the destination half-deleted
- **Category**: durability / atomicity
- **Confidence**: high
- **Location**: port.rs:209-249 (skill), :325-342 (memory), :451-465 (mcp)
- **Today**: Skill apply does `write_backup → remove_dir_all(dest) → copy_dir_recursive`. Crash between leaves user without destination. MCP write does `fs::write(dest_path, merged)` which on most filesystems truncates-then-writes — crash mid-write leaves zero-byte/partial JSON.
- **Proposed**: `write_atomic` helper for files (tmp + fsync + rename). For skill (directory) ports, swap pattern: `copy_dir_recursive(src, dest.with_extension("port-tmp"))`, `rename(dest, dest.with_extension("port-old"))`, `rename(dest.with_extension("port-tmp"), dest)`, `remove_dir_all(dest.with_extension("port-old"))`. On Windows `rename` over existing dir fails; gate `cfg(windows)` and use move-old-aside path explicitly.

### F-R9-3 — `json_to_toml` silently corrupts MCP env values when a JSON `null` round-trips
- **Category**: correctness
- **Confidence**: high
- **Location**: port.rs:615-640
- **Today**: `serde_json::Value::Null` mapped to `toml::Value::String(String::new())`. `read_mcp_from` skips entries that fail to deserialize — single `"env": { "FOO": null }` quietly drops the entire server entry rather than warning.
- **Proposed**: In `read_mcp_from`, replace ignore-on-error with explicit `match` that pushes error name into `messages`. For `json_to_toml`, drop null-arm; return `Result`; propagate so a null in args/env aborts port with clear message. Per project rule #3 — silent server drop is crap.

### F-R9-4 — User-scope MCP port direction crosses scopes incoherently
- **Category**: semantics
- **Confidence**: medium
- **Location**: port.rs:391-408
- **Today**: When `user_scope: true`, source is `~/.claude/settings.json`. But Claude Code also supports `~/.claude.json` (legacy), `.claude/settings.local.json`. Codex has both `~/.codex/config.toml` and `~/.config/codex/config.toml` on Linux (XDG). Code picks one path, never tells user which.
- **Proposed**: Centralize in `ScopeResolver` enum: `Project { dir }`, `User`, `Managed`. Surface resolved path in `messages` always. Honor XDG_CONFIG_HOME on Linux. Fall back to `~/.claude.json` if `~/.claude/settings.json` missing.

### F-R9-5 — Skill port's hardcoded `99` rename-suffix ceiling is arbitrary
- **Category**: code quality
- **Confidence**: low
- **Location**: port.rs:160-162
- **Proposed**: Raise to `u32::MAX` or replace numeric suffix with timestamp suffix `{name}-{ts}`. Per project rule #3 — ask user.

### F-R9-6 — TAP listener leaks the OS port across stop-then-start cycles
- **Category**: resource leak
- **Confidence**: medium
- **Location**: tap_server.rs:152-292
- **Today**: Listener owned by spawned thread. On stop, `wake_tap_listener` connects to `127.0.0.1:port` to unblock `accept`. Race window: second connection between wake-connect and re-accept could land between teardown.
- **Proposed**: Use `socket2::Socket::from(listener).shutdown(Shutdown::Both)` from `stop_session`/`stop_all`. Sentinel marker line for wake. Sets non-blocking via `set_nonblocking(true)`.

### F-R9-7 — `set_active_client` holds `state.lock()` while the read loop runs
- **Category**: lock contention / latency
- **Confidence**: medium
- **Location**: tap_server.rs:57-78, 184-186, 206-208, 234-244
- **Today**: `should_stop` called inside drain loop AND on every outer `read_line`. Each call locks global `Arc<Mutex<TapServerState>>`. With N concurrent sessions, lock becomes throughput ceiling.
- **Proposed**: Per-session `Arc<AtomicBool>` cloned into thread at spawn. Thread checks `flag.load(Relaxed)` — wait-free. State map keeps only metadata (port, AtomicBool handle, optional client clone for shutdown).

### F-R9-8 — Read errors on the TAP socket re-classify as warnings even when the socket simply closed
- **Category**: log noise
- **Confidence**: low
- **Diff**: small
- **Location**: tap_server.rs:204-220, 234-247
- **Proposed**: Match on `err.kind()`: `ConnectionReset | BrokenPipe | UnexpectedEof | ConnectionAborted` → LOG level "client disconnected"; else WARN.

### F-R9-9 — Metrics collector is always-on with no opt-out and a fixed 1000 ms tick
- **Category**: performance / power
- **Confidence**: medium
- **Location**: metrics.rs:14, 72-202
- **Today**: `POLL_INTERVAL = 1000ms` hardcoded. Always wakes. When no PIDs tracked, still wakes every 1s, locks `ActivePids`, finds empty, sleeps. 86,400 wakeups/day on battery for nothing. `system.refresh_processes_specifics(ProcessesToUpdate::All, ...)` scans /proc on Linux every second.
- **Proposed**:
  1. Idle path: `Condvar::wait_timeout(60s)` + `register_active_pid` notifies. Wakes only when user opens a session.
  2. Take `POLL_INTERVAL` from settings (`metrics.poll_ms`, default 1000). Adaptive.
  3. Refresh only tracked roots + descendants, not ALL via `ProcessesToUpdate::Some(&[Pid])`.

### F-R9-10 — `sum_descendants` BFS uses unbounded queue
- **Category**: robustness
- **Confidence**: low
- **Location**: metrics.rs:215-256
- **Proposed**: Hard cap on `queue.len()` and `visited.len()` (~4096); emit single WARN `metrics.descendants_truncated` event when hit.

### F-R9-11 — Trait-based platform abstraction would eliminate /proc parser surprises
- **Category**: architecture
- **Confidence**: medium
- **Location**: metrics.rs:74-203
- **Today**: Single concrete `sysinfo::System`. No telemetry on whether sysinfo failed `/proc/<pid>/status` on Linux or `Access Denied` on Windows via WMI.
- **Proposed**: `trait MetricsBackend { fn refresh(&mut self, pids: &[u32]) -> RefreshResult; ... }`. `Counters` struct tracks errors, exposed via `metrics.diagnostics` Tauri command. Per CLAUDE.md "implement telemetry first" — this is the canonical case.

### F-R9-12 — Weather poll loop has no jitter, no rate-limit, busy-loops on persistent failure
- **Category**: rate-limit / robustness
- **Confidence**: high
- **Location**: weather/mod.rs:204-246
- **Today**: If `fetch_open_meteo` returns `Err`, sleeps 60s and retries forever with no backoff. No jitter — every Code Tabs instance with `cf-ipcountry: AU` running for 30 min hits Open-Meteo at same wall-clock minute.
- **Proposed**:
  1. Exponential backoff on Err: `min(60s * 2^retries, 30min)`, capped, with 0-30s jitter.
  2. 0-60s jitter on success-path 30-min sleep.
  3. Make POLL_INTERVAL/RETRY_INTERVAL configurable.
  4. Skip network call if cached payload < poll_minutes old AND country unchanged.

### F-R9-13 — Weather coordinates list is hard-coded and incomplete; non-default users see Sydney
- **Category**: correctness
- **Confidence**: medium
- **Location**: weather/mod.rs:45-101
- **Today**: ~50 countries hardcoded, fall-through is Sydney. `_ => (-33.87, 151.21, "Sydney")` shares coordinates with `AU`, so renderer can't distinguish "fallback" from "real Australia".
- **Proposed**: Ship `country-capitals.json` resource (197 entries), loaded via `include_str!`. On unknown code, set label to `"Unknown ({cc})"` and skip fetch.

### F-R9-14 — `weather.json` cache write is non-atomic
- **Category**: durability
- **Confidence**: low
- **Location**: weather/mod.rs:182-192
- **Proposed**: Reuse `write_atomic` helper from F-R9-2.

### F-R9-15 — `set_country` runs on every proxy response — unbounded mutex acquisitions
- **Category**: performance
- **Confidence**: low
- **Location**: weather/mod.rs:113-124, called from proxy/mod.rs:700
- **Proposed**: `OnceLock<ArcSwap<Option<String>>>` or fast path `if guard.as_deref() != Some(upper.as_str())` to avoid writing identical values.

## Cross-cutting concerns

- **Atomic writes everywhere or nowhere** — three files write user-visible state non-atomically. One shared `fs_atomic::write(path, bytes)` helper.
- **Always-on background work needs an off switch** — `metrics::spawn_collector` and `weather::init` both start at boot, run forever, no setting to disable. Per project rule #6.
- **Telemetry before hypotheses** — counters per subsystem, expose via diagnostic Tauri commands.
- **Win+Linux symlink path in port_memory** — Windows `symlink_file` requires Developer Mode/admin. Surface clear error.
