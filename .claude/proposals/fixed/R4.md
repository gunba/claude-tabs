## Findings

### F1 — `parse_features` mishandles multi-word `Stage` strings (real bug)
- **Category**: correctness
- **Confidence**: high
- **Diff**: small (~15 LOC)
- **Location**: codex_cli.rs:320-349 + test at 1131
- **Today**: `parse_features` does `trimmed.split_whitespace()` and reads `parts[0]=name parts[1]=stage parts[2]=state`. Real Codex source (codex-rs/cli/src/main.rs:701-709 `stage_str`) emits `"under development"` for `Stage::UnderDevelopment` — TWO words. Feature `js_repl  under development  false` is parsed as `name="js_repl" stage="under" state="development"` and `enabled` becomes false regardless. State matcher only accepts `"on"|"enabled"|"true"`; real value is `true`/`false` (codex-rs/cli/src/main.rs:1203). `"on"|"enabled"` literals never appear and are dead code. **Test fixture is fictitious**: `parse_features_extracts_tabular_rows` asserts a header row real binary never prints.
- **Proposed**: Split on `\s{2,}` (column emitter pads with two-space separators per L1203). Update state matcher to accept `"true"`/`"false"`. Drop `"on"`/`"enabled"`. Replace bogus header in test fixture.

### F2 — `parse_mcp_list` is wrong about actual `codex mcp list` shape (high-impact bug)
- **Category**: correctness
- **Confidence**: high
- **Diff**: medium
- **Location**: codex_cli.rs:379-428, 430-438 + tests at 1142, 1154
- **Today**: Whitespace-tabular branch assumes 3 columns. Real Codex output (codex-rs/cli/src/mcp_cmd.rs:555-710) emits **two separate sections** — stdio rows (7 columns) and http rows (5 columns). Args is whitespace-joined (multi-word). Status can be `disabled: <reason>`. **Codex supports `--json` already** (mcp_cmd.rs:496) returning clean JSON.
- **Proposed**: Stop parsing tabular. Use `--json`. Update `CodexMcpServer` to match Codex's actual JSON shape. Delete tabular fallback. Drop `parse_mcp_list_handles_text` test — fixture is fictional.

### F5 — `generate_codex_session_title` orphans Codex subprocess on timeout
- **Category**: correctness / resource-leak
- **Confidence**: high
- **Diff**: tiny (1-2 lines)
- **Location**: codex_cli.rs:988-1012
- **Today**: `tokio::process::Command::new(&bin) ... .output()`. Wrapped in `tokio::time::timeout(60s, exec)`. On timeout outer future is dropped, but `tokio::process::Command::output()` does not set `kill_on_drop`, so spawned `codex exec` keeps running orphaned. Tempfile gets deleted **while orphaned codex may still be writing**.
- **Proposed**: Add `.kill_on_drop(true)` to `tokio::process::Command` builder. Distinguish timeout error from spawn error in surfaced string.

## Cross-cutting concerns

1. **`run_codex` vs. `run_claude_cli` divergence**: Claude's variant (commands/cli.rs:145) does Bun-runtime workarounds. `run_codex` is 15 lines of stock `Command::output()`. **No shared "spawn-with-CREATE_NO_WINDOW + label" helper** exists. With ~9 spawn sites in `commands/`, a `crate::process::sync_output(bin, args, label)` helper would consolidate. Flag to **R7** and **R10**.

2. **Binary scanning is duplicated**: this module's `discover_codex_slash_commands_sync` (memchr probes) + `discovery/codex.rs::mine_codex_env_var_names` and `mine_schema_from_binary` all walk same Codex native binary. A shared `BinaryView` helper that reads-once would share the 500 MiB cap. Flag to **R10**.

3. **`#[tauri::command]` async wrapper boilerplate**: 11 of 12 Tauri commands here are `tauri::async_runtime::spawn_blocking(fn_sync).await.map_err(...)`. Same in commands/cli.rs and commands/data.rs. Macro could eliminate. Flag to **R10**.
