# R5: proxy/mod.rs

## Files in scope
- src-tauri/src/proxy/mod.rs (1246 lines)

## Findings

### F1 — Per-request regex compilation in async context
- **Category:** performance / concurrency
- **Confidence:** high
- **Diff size:** medium (~80 LoC + small struct)
- **Location:** `apply_rules_to_text` L941-965 (called from `rewrite_system_prompt_in_body` L856 and `rewrite_openai_instructions_in_body` L888); rules are also defensively recompiled in `update_system_prompt_rules` L226-237.
- **Today:** Every proxied Anthropic / OpenAI / ChatGPT request that reaches a rewritable endpoint enters `apply_rules_to_text`, which iterates `rules` and calls `regex::Regex::new(&pattern)` once per rule per request inside the async tokio task that owns the connection. The same pattern is also compiled and discarded by `update_system_prompt_rules` for validation. With a moderate rule set (e.g. 20 patterns) and an active session firing several requests per second, this multiplies straight into hot-path CPU on tokio's runtime. The whole rewrite, including JSON re-serialization, is also synchronous CPU work in an async task.
- **Proposed:**
  1. Add `CompiledRule { id, replacement, regex: regex::Regex }` and a top-level `regex::RegexSet` for fast "does any rule match?" pre-check.
  2. Compile once in `update_system_prompt_rules`, store `Vec<CompiledRule>` in `ProxyInner` instead of (or alongside) `rules: Vec<SystemPromptRule>`. Pattern validation becomes free — successful compile is the validation.
  3. In the connection handler, skip the rewrite branch entirely when `RegexSet::is_match` returns no hits.
  4. Wrap the rewrite (`replace_all` + `serde_json::to_vec`) in `tokio::task::spawn_blocking` when body > 64 KiB so a slow regex against a 4 MiB body cannot stall other proxy connections on the same runtime worker.
- **Side effects:** `update_system_prompt_rules` becomes the single source of compilation. Existing tests still work via the same entry points. Tauri command surface unchanged. Match-counter increments stay on the connection task (cheap HashMap update).

### F2 — Rule loop applies every rule with re-quoted flags every iteration
- **Category:** performance / clarity
- **Confidence:** high
- **Diff size:** small (~30 LoC)
- **Location:** `apply_rules_to_text` L946-964
- **Today:** Inner loop computes `inline_flags` filter, allocates a `String`, and `format!`s `(?{}){}` every request and every rule even when the pattern doesn't match. There's no short-circuit on empty input. Invalid patterns silently fail per request via `if let Ok(re) = regex::Regex::new(&pattern)`. Final `replaced != result` is an O(n) string compare to detect a change `replace_all` already knows about via `Cow`.
- **Proposed:** With F1's `CompiledRule` precompiled, replace the loop body with the compiled `regex.replace_all(...)` and use `Cow::Borrowed` vs `Owned` to detect a real change instead of an O(n) string equality. (Not "short-circuit on first match" — multiple rules can validly fire in sequence — just eliminating per-request format/compile.)
- **Side effects:** Identical observable behaviour for valid rules. Invalid patterns are rejected at config time; runtime silent-fail branch goes away. Tests stay green.

### F3 — Mutex held across rule rewrite + JSON parse + traffic log writes
- **Category:** concurrency / latency
- **Confidence:** medium-high
- **Diff size:** small
- **Location:** `handle_connection` L129-132, L538-543, L567-573; `write_traffic_entry` L845-851.
- **Today:** Single `Arc<Mutex<ProxyInner>>` couples `default_client` clone, `rules` clone, `should_log` lookup, match-counter increments, traffic-log file lookup, and the actual log-write path (`writeln!` + `flush` under the same mutex — disk IO under the proxy mutex). Combined with F4 (per-request flush), every logged request blocks every other proxy task that wants to peek at state.
- **Proposed:** Split into independent locks:
  - `Arc<RwLock<RuleSet>>` (rules + RegexSet); match counters as `Arc<RwLock<HashMap<String, AtomicU64>>>` so increments are lock-free after the entry exists.
  - `Arc<Mutex<TrafficLogs>>` (file-handle map only).
  - `default_client` directly cloneable on a top-level field — `reqwest::Client` is `Arc`-cheap, no lock needed to read.
- **Side effects:** Slightly more types in `ProxyInner`. `get_rule_match_counts` becomes a snapshot read. Removes coupling between traffic-log writes and rule updates.

### F4 — Per-request `BufWriter::flush` defeats buffering
- **Category:** performance
- **Confidence:** very high
- **Diff size:** tiny (delete a line) or small (interval flush task)
- **Location:** `write_traffic_entry` L848-849
- **Today:**
    ```
    let _ = writeln!(writer, "{}", line);
    let _ = writer.flush();
    ```
   `BufWriter` + flush-every-line is a `File::write` with extra ceremony. For a streaming SSE response + per-entry flush, that's roughly one syscall per request to `traffic.jsonl`. NTFS makes this even more expensive than ext4.
- **Proposed:** Two options:
  1. **Drop the per-write flush.** `BufWriter` flushes on `Drop`, and `stop_traffic_log` already calls `writer.flush()` (L382). One missing line on a hard kill is acceptable for a debug log. One-line fix.
  2. **Interval flusher** via `tokio::time::interval` (every ~250ms or on stop signal). Bounded loss window.

  Recommend (1) for simplicity.
- **Side effects:** Crash-loss window grows from near-zero to BufWriter capacity (8 KiB default) of buffered entries. Acceptable for a debug log.

### F5 — Read loop allocates a header buffer string on every iteration
- **Category:** performance
- **Confidence:** high
- **Diff size:** small
- **Location:** `handle_connection` L500-524
- **Today:** `String::from_utf8_lossy(&buf[..hend])` runs every iteration of the read loop just so `extract_content_length(&str)` can match against a string. With 8 KiB chunks, a 100 KiB body re-allocates ~13 times before the body completes.
- **Proposed:** Add `extract_content_length_bytes(&[u8])` doing case-insensitive byte matching for `b"content-length:"` (`memchr` is already a dep — Cargo L71). Headers are ASCII-only per RFC 7230 so byte matching is correct. Only re-scan when we just located `hend`.
- **Side effects:** None. Pure perf win. Existing `extract_content_length` test stays via a thin wrapper.

### F6 — Body buffered to 50 MiB before any forwarding, even for non-rewritable endpoints
- **Category:** memory / latency
- **Confidence:** high
- **Diff size:** medium
- **Location:** `handle_connection` L500-524 (read loop); L597+ (build req).
- **Today:** Whole body buffered into a `Vec<u8>` (cap 50 MiB) before forwarding, regardless of whether the endpoint is rewritable. For non-rewritable paths (`/backend-api/codex/account` GETs that Codex hits on session boot, etc.) this wastes memory and latency.
- **Proposed:** Branch on `rewritable_prompt_endpoint`:
  - **Rewritable path:** read to memory (current behaviour), apply rules, forward.
  - **Pass-through path:** read headers only, then stream `stream` → upstream via `reqwest::Body::wrap_stream(...)`. Traffic logging records `req_len` from `Content-Length` header (or `null`) instead of holding the body.
- **Side effects:** Big files via the proxy stop being capped at 50 MiB for non-rewritable endpoints; for rewritable endpoints the cap stays. Streaming requires `stream` feature on reqwest — already enabled in Cargo (L58). Frontend (PromptsTab) only reads match counts, doesn't care.

### F7 — Upstream resolver dispatches via if/else with a default-to-Anthropic fallback
- **Category:** correctness / maintainability
- **Confidence:** medium
- **Diff size:** small
- **Location:** `resolve_upstream` L481-491; called once at L547.
- **Today:** Default fallback to `UpstreamKind::Anthropic` at L489 means a typo like `/v2/messages` quietly hits `https://api.anthropic.com/v2/messages` and gets an upstream 404, instead of a clearer 400/404 from the proxy.
- **Proposed:** Replace with an ordered slice of `(matcher_fn, UpstreamKind)`. Return `Option<UpstreamKind>` and respond 404 at the caller for misses:
    ```
    const ROUTES: &[(fn(&str) -> bool, UpstreamKind)] = &[
        (is_anthropic_endpoint, UpstreamKind::Anthropic),
        (is_chatgpt_codex_path,  UpstreamKind::ChatGpt),
        (is_openai_v1_path,      UpstreamKind::OpenAi),
    ];
    ```
- **Side effects:** Behavioural change — previously misrouted unknown paths now return a proxy error instead of an upstream 404. Better error surface, but worth a config flag in case Claude Code or Codex hits an undocumented path. Existing positive tests at L1168-1184 stay; add one negative case.

### F8 — Zero handling of upstream gzip / brotli / deflate response when traffic-logging
- **Category:** correctness
- **Confidence:** high
- **Diff size:** small
- **Location:** Response header forwarding L703-711; `decompress_if_gzip` L779-789.
- **Today:** Proxy forwards client `Accept-Encoding` to upstream unchanged (header strip loop at L601-607 only removes `host` and `content-length`). Upstream returns compressed body; proxy correctly forwards encoded bytes to the client, but stores encoded bytes in `resp_buf` and `decompress_if_gzip` only magic-bytes-checks for gzip (`0x1f 0x8b`). Brotli, deflate, and zstd responses get logged as garbled binary. Cloudflare-fronted endpoints (`api.anthropic.com`, `chatgpt.com`) negotiate `br` first when offered.
- **Proposed:**
  1. **Strip `Accept-Encoding` from the proxied request** (Anthropic doesn't compress requests by default; Codex might — guarantees rule matcher sees plaintext).
  2. **Replace `decompress_if_gzip` with `decompress_response(content_encoding, bytes)`** that dispatches on the captured `content-encoding` header. Use `flate2` (already a dep) for gzip/deflate, leave brotli/zstd as `"(brotli body, not decoded)"` until the dep is added. Honest > silent corruption.
- **Side effects:** Increased response bandwidth to the proxy (uncompressed). Localhost only — no real cost. Eliminates garbled rule matching and unreadable logs.

### F9 — Hardcoded 300 s total timeout, applied uniformly
- **Category:** robustness
- **Confidence:** medium
- **Diff size:** tiny
- **Location:** `build_plain_client` L61-66
- **Today:** Single `timeout(Duration::from_secs(300))` is the *total* request timeout. SSE streams that legitimately run past 5 minutes (long Claude reasoning traces) get killed.
- **Proposed:**
    ```
    .read_timeout(Duration::from_secs(180))   // per-chunk idle, correct for SSE
    .connect_timeout(Duration::from_secs(10))
    .pool_idle_timeout(Duration::from_secs(90))
    ```
   Drop total `timeout`. `read_timeout` requires reqwest 0.12+ — already on 0.12 (Cargo L58).
- **Side effects:** Long streams over 5 min stop being killed. A wedged upstream takes 3 min of silence to detect (vs. today's 5 min absolute cap on the unhappy path).

### F10 — `extract_session_id` allocates two `String`s per request
- **Category:** performance
- **Confidence:** medium
- **Diff size:** trivial
- **Location:** L969-978
- **Today:** Two heap allocations on every request just to split out the optional `/s/{id}/` prefix.
- **Proposed:** Return `(Option<&str>, &str)`. Only `to_string()` the session id when we know we'll log this request.
- **Side effects:** Small lifetime juggling in `handle_connection`.

### F11 — `parse_request` re-walks the buffer for header end already located in the read loop
- **Category:** performance
- **Confidence:** medium
- **Diff size:** small
- **Location:** `find_header_end` L980-987; called twice (L510 in read loop, again in `parse_request` L1000).
- **Today:** Manual O(n) byte-quad scan, run twice.
- **Proposed:** Pass `hend` from the read loop into `parse_request`. Switch to `memchr::memmem::find(buf, b"\r\n\r\n").map(|i| i + 4)` (memchr already a dep, SIMD on x86_64/AArch64).
- **Side effects:** None. `test_find_header_end` keeps using the pure function.

### F12 — Cf-IPCountry weather forwarding lives inside the proxy hot path

## Cross-cutting concerns

- **Async-friendliness:** F1 (precompiled regex + spawn_blocking on big bodies) and F3 (split locks + atomic counters) are the two changes that make this module behave under contention. Rust's `regex` engine is linear-time (Thompson NFA) so a "slow" pattern is bounded — but it's still synchronous CPU under tokio's worker pool.
- **Windows + Linux:** No platform-specific surfaces affected.
- **No-cache constraint:** F1's `RegexSet` and `CompiledRule` are precomputed acceleration structures over inputs (rules) that change rarely — the same pattern as keeping a parsed config struct rather than re-parsing JSON each read. No memoization of request-derived data.
- **No arbitrary delays:** F4(2)'s tokio interval is event-driven. F9's `read_timeout` replaces an arbitrary 300 s wall clock with per-chunk idle timeout — strictly less arbitrary.
- **Frontend impact:** Tauri command shapes unchanged. PromptsTab 2-second poll (CM-32) compatible.
- **Crap to delete (per CLAUDE.md guideline #3 — ask user first):**
  - `if let Ok(re) = regex::Regex::new(&pattern)` runtime swallow at L954 (F1 makes it impossible).
  - `replaced != result` O(n) compare at L956 — `Cow` already tells us (F2).
  - `.unwrap_or_else(|_| body.to_vec())` at L883 / L910 — re-serialization of a `Value` we just successfully parsed should not silently fall back; either `expect` it or panic-with-log.
  - Default-to-Anthropic fallback at L489 (F7) — looks like a leftover from when Anthropic was the only upstream.
  - Defensive double-compile in `update_system_prompt_rules` (L226-237).

## Confidence summary

Highest-leverage (do first): **F1, F4, F8, F6, F3** — F1 and F8 fix correctness issues (silent invalid regex; garbled rule matching against compressed bodies). F4 is a one-line perf win. F6 caps memory + latency for non-rewritable endpoints. F3 unblocks contention under load. Solid proposal set; nothing low-confidence is large-diff.
