## Findings

### F3-01 — Pervasive shallow Map clone-on-write multiplies allocation cost
- **Category**: performance / architecture
- **Confidence**: high
- **Diff**: medium-large
- **Today**: sessions.ts performs `const next = new Map(s.someMap); next.set(...); return { someMap: next };` in 12 places. With `subagents`, `skillInvocations`, `commandHistory`, `processHealth`, `pidToSessionId`, `inspectorOffSessions`, `trafficRecording`, a single hot tap event triggers one Map clone. Across 8 sessions, parent reference graph reallocates 8 times per tick.
- **Proposed (a)**: Move `subagents`, `skillInvocations`, `commandHistory`, `processHealth`, `inspectorOff`, `trafficRecording` onto each `Session` (or parallel `Record<string, SessionExtras>`). Map churn collapses to 1× per write. Selectors compile down to `s.sessions.find(x => x.id === id)?.subagents`.
- **Proposed (b)**: Adopt immer. `set(produce(s, draft => { draft.subagents.set(id, [...]); }))`. Removes ~150 LOC of manual clone-set-return triplets.
- Recommend (a) — encodes invariant ("a session has skills"), reduces type surface.

### F3-02 — `pidToSessionId` is the only Map keyed by something other than sessionId
- **Category**: design
- **Confidence**: medium
- **Today**: PID-keyed (the metric poller routes by OS PID). Frontend only consumes `processHealth.tree`, never queries `pidToSessionId.get` from rendering.
- **Proposed**: Move pidToSessionId resolution entirely into Rust backend metrics module — emit `sessionId` directly with metric.

### F3-03 — `seenToolNames` / `seenEventKinds` are write-only telemetry
- **Category**: dead-code
- **Confidence**: medium
- **Diff**: small
- **Today**: Both are Sets fed by `useTapEventProcessor`, consumed by nothing in render code.
- **Proposed**: Either move to a debug ring buffer (with TTL/cap — `dlog`'s pattern), or delete. Per CLAUDE.md ("crap should be deleted"), worth check-or-justify. **Ask user first.**

### F3-04 — `activity.ts` kind-precedence is a setter invariant scattered across 60 lines
- **Category**: design / latent-bug
- **Confidence**: high
- **Diff**: medium
- **Location**: `addFileActivity` L144-241
- **Today**: [AP-03] (created>modified>read>searched) and [AP-06] (main-agent search wins) implemented inline twice — once for turn dedupe (L176-201), once for `allFiles` (L203-225). Logic identical in shape but not in code — textbook invariant-drift bug. "Don't downgrade modified to read" is encoded only in turn branch; "don't downgrade real-kind to searched" is encoded only in allFiles branch.
- **Proposed**: Single function:
  ```ts
  function mergeKind(prev: FileChangeKind, next: FileChangeKind): FileChangeKind | null
  function mergeAgent(prev: FileActivity, next: FileActivity): FileActivity | null
  ```
  Both branches reduce to `const merged = merge(prev, entry); if (merged) write merged`.

### F3-05 — `evictOldTurns` / `MAX_TURNS=50` violates "no caching" — re-examine
- **Category**: performance / CLAUDE.md compliance
- **Confidence**: high
- **Diff**: small-medium
- **Today**: Eviction in `startTurn` runs `Object.entries(activity.allFiles).sort(...)` whenever cap exceeded. `shift()` on `activity.turns` is O(n). `recomputeStats` (L82-93) reconstructs full Record on every `addFileActivity` (quadratic in turn.files.length).
- **Proposed**: If cap genuinely necessary, ring-buffer / Deque shape. Otherwise, lift cap to panel selector and let store hold everything. `recomputeStats` should be derived state in panel or maintained incrementally.

### F3-06 — `tapEvents.ts` 744-LOC discriminated union: real-cost analysis before shrinking
- **Category**: design / TS compile perf
- **Confidence**: medium
- **Diff**: large (type-only)
- **Today**: 80 event variants. MISSED-HOOKS additions (`AsyncFileOp`/`BunFileOp`/`AbortSignal`/`FileWatch`/`TextDecoderChunk`/`EmitterEvent`/`EnvAccess`/`ConsoleOutput`/`SyncFileOp`/`TimerOp`/`BunOp`/`WebSocketOp`/`StreamOp`) appear in union but no reducer/consumer usage found.
- **Proposed**: Split:
  ```ts
  type TapEvent = StateShapingEvent | { kind: TelemetryKind; [k: string]: unknown };
  ```
  State reducers `switch` over `StateShapingEvent` only — type-checker no longer threads 80 narrow types through every consumer. **Measure first**: time `tsc --noEmit` before/after.

### F3-07 — `version.ts:downloadAndInstallAppUpdate` swallows progress events into a closure
- **Category**: correctness
- **Confidence**: medium
- **Diff**: small
- **Location**: L92-111
- **Today**: `contentLength` and `downloaded` are closure-locals updated by event callbacks. If `Started` arrives twice the second overwrites; if `Progress` arrives with `chunkLength` but `contentLength` is 0, progress bar never updates. Catch path resets state but happy-path between `Finished` and `relaunch()` doesn't reset on relaunch failure.
- **Proposed**: Move state into store; add try/finally for reset.

### F3-08 — `weather.ts` has a module-scope mutable
- **Category**: testability / HMR / clarity
- **Confidence**: medium
- **Diff**: trivial
- **Location**: L32 `let listenerInstalled = false`
- **Today**: HMR breaks (preserves listenerInstalled but loses listener); test isolation broken (vitest doesn't reset module state); state should live in store (`initialized: boolean` already exists at L41).
- **Proposed**: Gate on `if (get().initialized) return;` and set after wiring listener.

### F3-09 — `isSessionIdle` precedence inversion bug risk
- **Category**: correctness
- **Confidence**: medium
- **Diff**: small
- **Location**: types/session.ts:14-19
- **Today**: `isSessionIdle("idle" | "interrupted") === true`. `isSubagentActive` returns true for everything *except* `dead | idle | interrupted`. Combined: `starting`, `error`, `actionNeeded`, `waitingPermission` are all "active". `error` being "active" is jarring.
- **Proposed**: Verify with consumers; document decision; add unit tests covering all 9 states.

### F3-10 — `closeSession` cleanup is a footgun
- **Category**: design / future-bug-class
- **Confidence**: high
- **Diff**: small (after F3-01)
- **Location**: L274-300
- **Today**: Reads previous state in three different ways then hand-deletes from 6 Maps + 1 Set. If a future Map is added, developer forgets `delete` here, entry leaks until app restart.
- **Proposed**: After F3-01 (a), cleanup is one line: `sessions.filter(x => x.id !== id)`. Or add `purgeSessionId(state, id)` helper.

### F3-12 — `tapEvents.ts` — `[key: string]: unknown` on `TapEntry` defeats the union

### F3-13 — Session selectors in components recompute Map.get on every render
- **Category**: performance
- **Confidence**: medium
- **Diff**: small (per call-site)
- **Today**: Patterns like `useSessionStore((s) => sessionId ? s.commandHistory.get(sessionId) : undefined) ?? [])` (CommandBar.tsx:29) return fresh array literal `[]` on every render. Zustand uses `Object.is`; fallback breaks memoization.
- **Proposed**: Hoist `const EMPTY: CommandHistoryEntry[] = []` to module scope.

### F3-14 — Test coverage gap: no tests for `processTreeMetricsEqual`
- **Category**: test gap
- **Confidence**: low
- **Diff**: small
- **Location**: sessions.ts:98-116
- **Today**: Hand-written deep-equality with index-based `topChildren` comparison. If Rust ever returns `topChildren` in unstable order, equality returns false and triggers infinite re-renders.
- **Proposed**: Add test: same children in different order should equal (or document ordering is contractual).
