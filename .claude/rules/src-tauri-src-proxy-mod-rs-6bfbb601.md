# src-tauri/src/proxy/mod.rs

Tag line: `L<n>`; code usually starts at `L<n+1>`.

## Slimmed Proxy

- [SP-01 L3] Slim prompt-rewrite proxy at src-tauri/src/proxy/mod.rs forwards POST /v1/messages to https://api.anthropic.com and OpenAI Responses traffic (POST /v1/responses or /backend-api/codex/responses) to api.openai.com/chatgpt.com. It applies user-defined regex rules scoped by CLI: Claude rules rewrite Anthropic system text, Codex rules rewrite Responses prompt fields (top-level instructions, developer/system input_text blocks, and marked contextual user prompt blocks such as AGENTS.md and skill payloads) before forwarding; ordinary user input_text is not rewritten. Incoming request bodies with Content-Encoding (zstd, gzip, deflate, identity) are decoded once up front in handle_connection before model extraction / user-turn classification / rule application; the Content-Encoding header is stripped from the forwarded request on successful decode so the upstream receives plain JSON, and decode failures pass through with a proxy.request_decode_failed WARN event. It optionally tees request/response to per-session traffic.jsonl, classifies fresh user turns vs tool-result follow-ups, and emits user-turn-started-{sessionId} on UserTurn so the frontend response panel clears at real send time. No model translation or OAuth.
- [SP-02 L585] Per-request upstream resolver routes Anthropic endpoints (/v1/messages and /v1/complete), OpenAI /v1/* endpoints, and ChatGPT Codex /backend-api/codex/* endpoints through UpstreamKind. path_matches_endpoint accepts exact matches, ? query suffixes, or / subpaths; is_chatgpt_responses_endpoint is intentionally limited to /backend-api/codex/responses for rewrites. rewrite_openai_prompt_in_body applies enabled Codex-scoped prompt-rewrite rules to Responses json["instructions"], developer/system input_text content, and marked contextual user input_text blocks matching Codex AGENTS.md (# AGENTS.md instructions for ... </INSTRUCTIONS>) or skill (<skill>...</skill>) markers; ordinary user-role input_text is skipped. Traffic logs include an upstream label (anthropic|openai|chatgpt) on request/response/error events.

- [SP-03 L698,845,1040] Request body decompression (decompress_request_body) handles zstd, gzip, deflate, and identity Content-Encoding values (case-insensitive, comma-separated chains processed right-to-left), returning Some(decoded) on success. handle_connection invokes it immediately after parse_request: if the body decodes, the proxy uses the plain JSON for model extraction, classify_user_turn, rule application, and the upstream request, dropping the Content-Encoding header from the forwarded headers. On decode failure (unknown encoding or malformed bytes) the original bytes pass through unchanged with a proxy.request_decode_failed WARN event carrying rawPath/contentEncoding/byteLength so misbehaving clients are surfaced instead of 502'd. zstd handling exists because Codex CLI compresses /backend-api/codex/responses bodies with zstd when CodexAuth::uses_codex_backend is true and the provider is_openai (our Code Tabs proxy provider is registered with name='OpenAI', so the path activates on every ChatGPT-auth Codex session); without this, serde_json::from_slice fails on the framed bytes and prompt-rewrite rules silently no-op.
  - Helper functions: decompress_zstd (zstd::stream::decode_all), decompress_gzip (flate2::GzDecoder), decompress_deflate (flate2 Zlib/Deflate). decompress_response was extended to handle zstd as well so the response side of traffic.jsonl is readable for ChatGPT-auth sessions. Unit tests at src-tauri/src/proxy/mod.rs cover zstd round-trip (test_decompress_request_body_zstd, test_decompress_response_zstd_by_header), no-encoding fallthrough, invalid zstd, and unknown encoding.

## Config Schema and Providers

- [CM-32 L428] Per-session rule match counters: ProxyState.rule_match_counts (HashMap<String, u64>) tracks how many times each prompt-rewrite rule (by rule ID) has matched a proxied request this session. Incremented in the proxy request handler for each matched rule ID. Exposed via get_rule_match_counts Tauri command (sync, returns clone of the map). PromptsTab polls this on a 2-second interval while the Rules sub-tab is active and displays match counts inline on each rule card: '0 matches' shows as 'never fired' (muted). Counter map is pruned to active rule IDs on each settings update.
